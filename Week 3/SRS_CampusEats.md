# SRS – ENGCE301 Lab Submission Portal

## 1. Introduction

### 1.1 Purpose (วัตถุประสงค์)
เอกสารฉบับนี้ระบุความต้องการ (Requirements) ของระบบระบบสั่งอาหารในมหาวิทยาลัย 
มีวัตถุประสงค์เพื่อ 

- ใช้เป็นแนวทางในการออกแบบ (Design) และพัฒนา (Development) ระบบสั่งอาหารออนไลน์, จองโต๊ะ, รีวิวร้านอาหาร ภายในมหาวิทยาลัย 
- ใช้เป็นเกณฑ์ในการทดสอบ (Testing) และตรวจสอบความสมบูรณ์ของระบบสั่งอาหารออนไลน์, จองโต๊ะ, รีวิวร้านอาหาร ภายในมหาวิทยาลัย 
- ใช้เป็นสัญญาระหว่างทีมพัฒนากับผู้มีส่วนได้ส่วนเสีย (Stakeholders) 

### 1.2 Scope (ขอบเขต)
ระบบสั่งอาหารในมหาวิทยาลัยเป็น มือถือแอป ที่มีวัตถุประสงค์เพื่อให้นักศึกษาสามารถสั่งอาหาร, จองโต๊ะ และรีวิวร้านอาหารภายในมหาวิทยาลัย

### 1.3 Actors & Roles

- **ผู้ใช้ทั่วไป** 
  - ผู้สร้าง LAB, แก้ไข LAB, ให้คะแนน/Feedback
- **นักศึกษา**  
  - ต้องการสั่งอาหาร จองโต๊ะ ดูเมนู ติดตามออเดอร์ รีวิวร้าน 
- **บุคลากร**
  - ใช้แอปเพื่อสั่งอาหารและจองโต๊ะคล้ายกับนักศึกษา 
- **ผู้ดูแลระบบ**
  - ต้องการตรวจสอบระบบ จัดการผู้ใช้ ร้านค้า เมนู รีวิวที่ไม่เหมาะสม 

### 1.4 Definitions, Acronyms
- **FR** – Functional Requirement 
- **NFR** – Non-Functional Requirement 
- **UI** – User Interface 
- **API** – Application Programming Interface 
- **SRS** – Software Requirements Specification 
- **Admin** – ผู้ดูแลระบบ

## 2. Overall Description

### 2.1 System Overview
ระบบ CampusEats – ระบบสั่งอาหารในมหาวิทยาลัย เป็นระบบแบบ Web Application ทำงานบน 
Web Browser รองรับทั้งคอมพิวเตอร์และอุปกรณ์พกพา   

### 2.2 Assumptions
- ผู้ใช้มี Internet Connection ที่เสถียร 
- ผู้ใช้มี อุปกรณ์ที่รองรับ Web Browser เช่น Notebook / Smartphone 
- ร้านค้าที่เข้าร่วมโครงการสามารถใช้งานระบบผ่านอินเทอร์เน็ต และมีอุปกรณ์สำหรับรับคำสั่งซื้อ 
- ระบบยืนยันตัวตนของมหาวิทยาลัย (RMUTL Account API) และฐานข้อมูลมหาวิทยาลัยพร้อมให้บริการตลอด

## 3. Functional Requirements (FR)

### FR-001: User Registration
**Description**  
ผู้ใช้สามารถลงทะเบียนเข้าใช้ระบบด้วยรหัสนักศึกษา รหัสผ่าน อีเมล และข้อมูลส่วนตัว 

**Details**
- ผู้ใช้ต้องกรอกรหัสนักศึกษา, รหัสผ่าน, อีเมล และข้อมูลส่วนตัว
- ระบบต้องตรวจสอบความถูกต้องของอีเมลและรูปแบบรหัสนักศึกษา
- ต้องตรวจสอบว่ารหัสนักศึกษายังไม่ถูกใช้ลงทะเบียนมาก่อน
- เมื่อสมัครสำเร็จ ผู้ใช้สามารถเข้าสู่ระบบได้

---

### FR-002: User Login 
**Description**  
ผู้ใช้เข้าสู่ระบบด้วย Student ID และ Password

**Details**
- ผู้ใช้เข้าสู่ระบบโดยกรอก Student ID และรหัสผ่าน
- ระบบตรวจสอบความถูกต้องของข้อมูลเข้าสู่ระบบ
- ระบบเปรียบเทียบรหัสผ่านกับค่า Hash ในฐานข้อมูล
- หากถูกต้อง ระบบออก JWT Token และนำผู้ใช้เข้าสู่ Dashboard

---

### FR-003: Update User Profile 
**Description**  
ผู้ใช้สามารถแก้ไขข้อมูลส่วนตัว เช่น เบอร์โทร อีเมล รูปโปรไฟล์ และข้อมูลติดต่อ 

**Details**
- ผู้ใช้สามารถแก้ไขข้อมูลส่วนตัว เช่น เบอร์โทร อีเมล รูปโปรไฟล์ และข้อมูลติดต่อ
- ระบบตรวจสอบความถูกต้องของรูปแบบอีเมลและข้อมูลที่แก้ไข
- ระบบตรวจสอบว่าอีเมลใหม่ไม่ซ้ำกับของผู้อื่น
- เมื่ออัปเดตสำเร็จ ระบบแสดงข้อความยืนยัน “อัปเดตข้อมูลสำเร็จ”

---

### FR-004: Browse Restaurants & Menu 
**Description**  
ผู้ใช้สามารถดูรายการร้านอาหารและเมนูพร้อมรายละเอียด 

**Details (แนะนำ)**  
- ผู้ใช้สามารถดูรายชื่อร้านอาหารทั้งหมดในระบบ
- ผู้ใช้สามารถเลือกดูเมนูของร้าน พร้อมราคา รูปภาพ และสถานะเมนู
- ระบบดึงข้อมูลร้านและเมนูที่อยู่ในสถานะ active จากฐานข้อมูล
- ผู้ใช้สามารถเลือกเมนูเพื่อไปสู่ขั้นตอนการสั่งอาหาร

---

### FR-005: Place Food Order 
**Description**  
ผู้ใช้ทำการสั่งอาหารในแอป

**Details**
- ผู้ใช้เลือกเมนูอาหารที่ต้องการสั่ง พร้อมจำนวนและวิธีชำระเงิน
- ระบบตรวจสอบสถานะเมนูว่าเปิดให้สั่งหรือไม่
- ระบบคำนวณราคารวมและบันทึกคำสั่งซื้อ
- ระบบสร้างหมายเลขคำสั่งซื้อและตั้งสถานะเริ่มต้น (เช่น Pending)

---

### FR-006: Track Order Status 
**Description**  
ผู้ใช้ติดตามสถานะคำสั่งซื้อแบบเรียลไทม์ 

**Details**
- ผู้ใช้สามารถระบุ Order ID เพื่อดูสถานะปัจจุบันของคำสั่งซื้อ
- ระบบดึงสถานะคำสั่งซื้อแบบเรียลไทม์จากฐานข้อมูล
- ระบบแสดงผลสถานะ เช่น Preparing, Delivering, Completed

---

### FR-007: Place Food Order 
**Description**  
ระบบตรวจสอบการชำระเงิน 

**Details**
- ระบบรับข้อมูล Payment ID เพื่อใช้ตรวจสอบการชำระเงิน
- ระบบเชื่อมต่อ API ของ Payment Gateway เพื่อยืนยันสถานะการจ่ายเงิน
- ระบบแสดงผลการชำระเงิน เช่น Success / Failed
- ระบบอัปเดตสถานะคำสั่งซื้อหลังตรวจสอบเสร็จ

---

### FR-008: Table Reservation 
**Description**  
ผู้ใช้จองโต๊ะกับร้าน 

**Details**
- ผู้ใช้กรอกวันที่ เวลา จำนวนคน และเลือกร้านที่ต้องการจองโต๊ะ
- ระบบตรวจสอบว่าร้านมีโต๊ะว่างตามเวลาที่เลือกหรือไม่
- หากมีโต๊ะว่าง ระบบบันทึกคำขอจองลงฐานข้อมูล
- ระบบแสดงสถานะการจอง เช่น Confirmed / Pending

---

### FR-009: Submit Review
**Description**  
ผู้ใช้ให้คะแนนและรีวิวร้านอาหารที่เคยสั่ง 

**Details**
- ผู้ใช้กรอกคะแนนและข้อความรีวิวร้านอาหาร
- ระบบตรวจสอบสิทธิ์ว่าผู้ใช้เคยสั่งอาหารจากร้านนั้นมาก่อน
- หากมีสิทธิ์ ระบบบันทึกรีวิวลงระบบ
- รีวิวจะถูกแสดงในหน้าร้านอาหารแบบเรียลไทม์

---

### FR-010: Restaurant Login
**Description**  
ร้านค้าเข้าสู่ระบบด้วย Restaurant ID ,Password 

**Details**
- ร้านค้ากรอก Restaurant ID และ Password เพื่อเข้าสู่ระบบ
- ระบบตรวจสอบข้อมูลเข้าสู่ระบบและตรวจสอบรหัสกับค่า Hash
- หากถูกต้อง ระบบออก JWT Token ให้ร้านค้า
- ร้านค้าถูกนำไปยัง Dashboard การจัดการร้านอาหาร

---

### FR-011: Restaurant Menu Management 
**Description**  
ผู้ใช้จองโต๊ะกับร้าน 

**Details**
- ร้านค้าสามารถเพิ่ม แก้ไข หรือลบเมนูอาหารได้
- ระบบตรวจสอบความถูกต้องของข้อมูลเมนู เช่น ราคา สถานะ รูปภาพ
- เมื่อบันทึกสำเร็จ ระบบอัปเดตเมนูในฐานข้อมูล
- เมนูที่แก้ไขจะแสดงผลทันทีให้ผู้ใช้เห็น

---

### FR-012: Restaurant Registration 
**Description**  
Admin จะสมัครบัญชีสำหรับการจัดการร้านค้าให้ 

**Details**
- Admin ทำการสมัครบัญชีร้านค้าใหม่ โดยกำหนด Restaurant ID และ Password
- ระบบตรวจสอบความแข็งแรงของรหัสผ่าน
- ระบบ Hash password และบันทึกข้อมูลร้านค้า
- ร้านค้าสามารถใช้บัญชีนี้เข้าสู่ระบบเพื่อจัดการเมนู

---

### FR-013: Admin Dashboard 
**Description**  
Admin ดูภาพรวมของสถิตการสั่งอาหาร ,สถิติการจองโต๊ะ และรีวิวของร้านอาหาร 

**Details**
- Admin สามารถดูสถิติรวมของร้านค้า เช่น จำนวนคำสั่งซื้อ การจองโต๊ะ และรีวิว
- ระบบดึงข้อมูลสถิติจากฐานข้อมูลของร้านค้า
- ระบบประมวลผลและสรุปผลออกมาเป็น Dashboard
- Dashboard แสดงข้อมูลแบบภาพรวมให้ Admin วิเคราะห์ได้ง่าย

## 4. Non-Functional Requirements (NFR)

### NFR-001 Page Load Time
- หน้าเว็บต้องโหลดเสร็จภายใน ≤ 2 วินาที

### NFR-002 API Response Time
- API Response Time ต้องไม่เกิน 500 ms

### NFR-003 Concurrent Users
- ระบบต้องรองรับผู้ใช้พร้อมกันอย่างน้อย 100 คน โดยไม่เกิดปัญหาด้านประสิทธิภาพ

### NFR-004 Password Security
- ใช้ bcrypt ในการ Hash รหัสผ่าน

### NFR-005 Session Management
- JWT Session Token ต้องมีอายุ ไม่เกิน 24 ชั่วโมง

### NFR-006 Data Encryption
- ระบบต้องใช้ HTTPS สำหรับการสื่อสารทั้งหมดเพื่อความปลอดภัย

### NFR-007 User Flow Simplicity
- ผู้ใช้ต้องสามารถสั่งอาหาร (ฟีเจอร์หลัก) ได้ภายใน ≤ 3 คลิก

### NFR-008 Language Support
- UI ต้องรองรับ ภาษาไทย / อังกฤษ

### NFR-009 Clear Error Messages
- ระบบต้องแสดง Error Message ที่เข้าใจง่าย พร้อมคำแนะนำวิธีแก้ไข

### NFR-010 System Availability
- ระบบต้องมี Uptime 99%

### NFR-011 Backup
- ระบบต้องทำการ Backup อัตโนมัติทุก 24 ชั่วโมง

### NFR-012 Recovery Time
- หากระบบเกิดปัญหาระดับทั่วไป ต้องสามารถ Recovery ภายใน ≤ 5 นาที

### NFR-013 Browser Compatibility
- รองรับเบราว์เซอร์ Chrome, Firefox, Safari, Edge

### NFR-014 Screen Resolution Support
- รองรับความละเอียด Desktop (1920×1080), Tablet (768×1024), Mobile (375×667)

### NFR-015 Theme Mode
- UI ต้องรองรับทั้ง Dark Mode และ Light Mode

### NFR-016 PDPA Compliance
- การเก็บข้อมูลส่วนบุคคลต้องเป็นไปตาม PDPA

### NFR-017 Admin Log Retention
- ต้องเก็บ Log ของผู้ดูแลระบบอย่างน้อย 90 วัน

### NFR-018 Sensitive Data Protection
- ห้ามแสดงข้อมูลสำคัญใน log เช่น password, token, payment info
